# 테스트는 문서다.

## 1. 테스트가 문서인 이유

잘 작성된 테스트 코드를 보면 프로덕션 기능을 이해할 수 있다. 다양한 테스트 케이스, 엣지 케이스를 통해 프로덕션 코드를 이해하는 시간과 관점을 보완할 수 있으며 코드를 작성하며 고민했던 개인의 경험을 모두의 자산으로 공유할 수 있다.

그렇다면 어떻게 테스트 코드를 문서처럼, 잘 작성할 수 있을까?

## 2. DisplayName을 섬세하게

메서드 이름만으로는 테스트가 무엇을 하고자 하는지 표현하는데에는 한계가 있다.

JUnit5에 추가된 @DisplayName 어노테이션을 사용하면 테스트 메서드가 어떤 역할을 하는지 명시할 수 있다.

### 2.1. DisplayName 잘 작성하는 법

- 명사의 나열보다 문장으로 작성하기
  - 음료 1개 추가 테스트 X
  - 음료를 1개 추가할 수 있다. O
  - "~테스트" 지양하기
- 테스트 행위에 대한 결과까지 기술하기
  - 음료를 1개 추가할 수 있다. X
  - 음료를 1개 추가하면 주문 목록에 담긴다. O
- 도메인 용어를 사용하여 풍부하게 표현하는 것이 좋다.
  - **특정 시간 이전**에 주문을 생성하면 실패한다. X
    - 특정 시간은 카페 키오스크에서 사용하는 도메인 용어가 아니다.
  - **영업 시작 시간** 이전에는 주문을 생성할 수 없다. O
    - 영업 시작 시간은 팀원 모두가 공유하고 있는 도메인 용어다.
- 테스트 현상을 중점으로 기술하지 않는다.
  - 특정 시간 이전에 주문을 생성하면 **실패한다**. X
    - "실패한다"는 테스트하고 하는 내용과 관계 없다.
  - 영업 시작 시간 이전에는 주문을 생성할 수 없다. O

## 2.2. BDD 스타일로 작성하기

BDD 스타일이란 Behavior Driven Development의 약자로 TDD에서 파생된 개발 방법이다. 함수 단위의 테스트에 집중하기보다, 시나리오에 기반한 테스트 케이스 자체에 집중하여 테스트한다(개발자가 아닌 사람이 읽어도 이해 가능한 레벨로 짜는 것이 좋다).

### 2.2.1. Given / When / Then

BDD 스타일로 테스트 케이스를 작성할 때는 Given / When / Then 형식을 사용한다.

- Given: 시나리오 진행에 필요한 모든 준비 과정(객체, 값, 상태 등)
- When: 시나리오 행동 진행
- Then: 시나리오 진행에 대한 결과 명시, 검증

이렇게 작성하면 DisplayName 또한 명확하게 작성할 수 있다.

인텔리제이의 live templates 기능을 활용하면 보다 편하게 테스트 코드 템플릿을 작성할 수 있다.

## 3. 키워드 정리

- 핵심 키워드
  - @DisplayName - 도메인 정책, 용어를 사용한 명확한 문장
  - Given / When / Then - 주어진 환경, 행동, 상태 변화
  - TDD vs BDD
- 추가 키워드
  - JUnit vs Spock
