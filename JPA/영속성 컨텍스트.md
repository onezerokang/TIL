# 영속성 컨텍스트(Persistence Context)

## 개요

- 영속성 컨텍스트란 서버와 데이터베이스 사이에서 엔티티를 임시로 저장하고 관리하는 논리적인 영역이다.
- 클라이언트의 요청이 들어오면 엔티티 매니저가 생성되는데, 영속성 컨텍스트도 이때 만들어진다.
- 생성된 엔티티 매니저를 통해 영속성 컨텍스트에 접근하고 관리할 수 있다.

## 영속성 컨텍스트의 이점

### 1차 캐시

- 영속성 컨텍스트 내부에는 1차 캐시가 있다.
- 엔티티를 em.find()로 조회하거나, em.persist()로 저장할 경우 해당 엔티티를 1차 캐시에 저장한다.
- 엔티티를 조회할 때 먼저 1차 캐시에서 엔티티를 찾고, 없을 경우 DB에서 조회한 뒤, 조회한 엔티티를 1차 캐시에 저장한다.
- 1차 캐시는 엔티티를 Key-Value로 저장하는데, Key는 엔티티의 @Id 값이 된다.
- 따라서 영속 상태의 엔티티는 반드시 식별자 값이 있어야 한다.
- 만약 식별자 생성 전략을 GenerationType.IDENTITY로 지정할 경우, 엔티티를 영속 상태로 만들기 위해 em.persist()를 호출하는 시점에 INSERT SQL이 데이터베이스에 전송하여 ID를 받아온다.

### 동일성 보장(애플리케이션 레벨에서 REPEATABLE READ 보장)

- em.find()로 같은 엔티티를 여러번 조회할 경우, 새로운 객체를 반환하는 것이 아닌, 동일한 객체를 반환한다.
- 이는 처음 조회 시, 1차 캐시에 값을 저장하고, 두번째부터는 캐시의 값을 가져오기 때문이다.
- 따라서 하나의 트랜잭션에서 같은 데이터를 여러번 조회할 때 다른 데이터를 가져오는 Non-Repeatable Read 문제가 발생하지 않는다.

### 트랜잭션으로 인한 쓰기 지연

- 영속성 컨텍스트 안에는 쓰기 지연 SQL 저장소가 있다.
- em.persist()가 호출되면 1차 캐시에 저장하고 엔티티를 분석해서 Insert 쿼리를 생성하고 쓰기 지연 SQL 저장소에 쌓아둔다.
- flush()가 호출되는 시점에 쓰기 지연 SQL 저장소에 쌓여있던 쿼리가 데이터베이스로 전송되고 이후에 트랜잭션이 커밋된다.
- 한번에 쓰기 쿼리를 보내기 때문에 JPA가 내부적으로 최적화를 할 수 있고 성능적으로 이점이 있다.

### 지연 로딩(Lazy loading)

- 하나의 엔티티를 조회할 때 관계가 맺어진 다른 엔티티를 모두 가져오면 성능 저하의 원인이 된다.
- 따라서 관계가 맺어진 엔티티를 프록시 객체로 받아온 뒤, 참조 객체에 접근하는 시점에 조회하는 지연 로딩을 지원한다.

## 변경 감지(Dirty checking)

- 엔티티를 변경하고, 트랜잭션이 커밋되면 별도로 저장 메서드를 호출하지 않아도 업데이트 쿼리를 보낸다.
- 1차 캐시안에는 PK(식별자), 엔티티, 스냅샷(영속 상태가 된 시점의 상태)가 저장된다.
- flush()가 호출될 때, 영속성 컨텍스트는 엔티티의 현재 상태와 스냅샷을 비교하여 변경이 감지된 경우, 해당 변경 사항을 반영하기 위해 업데이트 SQL을 생성하고 실행한다.

## 플러시(Flush)

- 영속성 컨텍스트의 변경 내용을 데이터베이스에 반영하는 것이다(쿼리 실행)
- 플러시를 발생시키는 방법
  - em.flush() 호출
  - 트랜잭션 커밋
  - JPQL 쿼리 실행
    - em.persist()로 1차 캐시에만 저장된 엔티티를 JPQL로 쿼리하려고 하면 가져올 수 없다.
    - 따라서 JPQL가 실행될 때, flush 되도록 설정하여, 엔티티를 DB에 반영하는 것이다.

## 엔티티의 생명주기

- 비영속(new/transient): 영속성 컨텍스트와 전혀 관계 없는 새로운 상태
- 영속(managed): 영속성 컨텍스트에 관리되는 상태
- 준영속(detached): 영속성 컨텍스트에서 관리되다가 분리된 상태
- 삭제(removed): 삭제된 상태

준영속 상태의 엔티티는 Dirty checking 같은 영속성 컨텍스트의 기능이 적용되지 않기에 주의해야 한다.
