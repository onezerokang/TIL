2. JPA 내부 동작 방식 이해하지 못하고 사용.

## SQL 중심적인 개발의 문제점

애플리케이션 개발 시 객체지향 언어로 개발한다.
데이터베이스는 관계형 DB를 많이 사용한다.

지금 시대는 객체를 관계형 DB에 보관하고 관리하는 시대.
SQL 중심적인 개발의 문제점 - 무한 반복, 지루한 코드: 자바 객체를 SQL로, SQL을 자바 객체로

객체에 필드 하나 추가되면 SQL문을 전부 바꿔야 한다.

**예시 코드 작성하기**

패러다임의 불일치
OOP는 추상화, 캡슐화, 정보은닉, 상속, 다형성 등 시스템의 복잡성을 제어할 수 있는 다양한 장치를 제공한다.

개발자가 SQL mapper 역할을 해야 한다.
객체 -> SQL로 변환 -> RDB에 저장.

객체와 rds의 차이

1. 상속: 객체 상속 관계 vs 데이터베이스는 상속관계는 없다. 슈퍼타입 서브 타입 관계가 그나마 비슷하다.
   - 생성: create으로 하나하나 가져와야 한다.
   - 조회: join sql 작성, 각각의 객체 생성, 상상만해도 복잡
   - 그래서 DB에 저장할 객체는 상속 관계 안쓴다.
   - 자바 컬렉션에서 생성하고 조회하는 것은 편하다.
2. 연관관계
   - 객체는 참조를 상요한다. member.getTeam()
   - 테이블을 외래키를 사용해서 JOIN ON M.TEAM_ID = T.TEAM_ID
   - 객체를 테이블에 맞추어 모델링
   - 객체다운 모델링은 아니다. 객체는 참조로 연관관계를 맺는다(id만 저장하지 않는다)
   - 하지만 rdb는 id를 저장하고 조인한다.
   - 자바 컬렉션에 관리한다고 생각해보자
3. 객체 그래프 탐색
   - 객체는 자유롭게 객체 그래프를 탐색할 수 있어야 한다. 참조로 계속 따라갈 수 있어야 한다.
   - SQL은 처음 실행할 때 탐색 범위가 결정된다.
   - 엔티티 신뢰 문제: 데이터를 DAO에서 가져왔을 때 다음 계층 존재 여부를 알 수 없다. 직접 까봐야 함
   - 그렇다고 모든 연관관계를 조인하면 성능 이슈 발생
4. 비교하기
   - SQL로 같은 데이터를 꺼내서 객체로 변환하여 비교하면 다른 인스턴스
   - 자바 컬렉션에서 꺼내서 비교하면 같은 인스턴스

객체답게 모델링 할수록 매핑 작업만 늘어난다.
객체를 자바 컬렉션에 저장 하듯이 DB에 저장할 수는 없을까?
JPA - Java Persistence API

## JPA 소개

JPA?

- Java Persistence API
- java ORM 표준
  - Object-relational mapping
  - ORM 프레임워크가 객체와 DB를 매핑
- JPA는 애플리케이션과 JDBC 사이에서 동작한다.
  - JPA에서 멤버객체를 저장할 때 멤버dao에게 넘기고 dao가 jpa에 저장해달라고 던지기만 하면 jpa가 분석하고 insert jql을 만들고 jdbc를 sql문을 만든다.
  - JPA가 쿼리를 만들고 jdbc API를 사용해서 쿼리를 날린다. 패러다임 불 일치 해결, entity 오브젝트를 만들어서 반환한다.

### jpa 역사

- EJB - 엔티티 빈(자바 표준) 복잡도 높고, 성능 안나와서 실용성이 떨어졌다. 개발자들이 사용하지 않음
- 하이버네이트(오픈소스): 내가 만들어도 이것보다는 낫겠다.
- JPA(자바 표준): 자바진영에서 게빈킹을 불러와서 자바 표준을 만들어달라고 요청 -> JPA(하이버네이트와 비슷하다)

JPA는 표준 명세다(인터페이스의 모음이다)

- JPA는 인터페이스의 모음
- JPA 2.1 표준 명세를 구현한 3가지 구현체: 하이버네이트, 이클립스링크, DataNucles

### 왜 써?

- sql 중심적인 개발에서 객체 중심으로 개발(객체 컬렉션 사용하듯이 db사용 가능)
- 생선성: jpa.persist(member)(영구 보관하다), jpa.find(), member.setName(), jpa.remove()
- 유지보수: 기존 필드 변경 시 모든 SQL 수정, jpa는 필드만 추가하면 됨, sql은 jpa가 처리
- 패러다임의 불일치 해결
  - 상속: jpa.persist()만 하면 나머진 JPA가 처리한다. 조회: 가져오기만 하면 알아서 조인한다.
  - 연관관계, 객체 그패트 탐색: jpa는 연관관계 저장, 객체 그래프 탐색이 가능하다.(지연 로딩)
- jpa와 비교하기

  - 동일한 트랜잭션에서 조회한 엔티티는 같음을 보장한다.
  - JPA의 성능 최적화 기능.
    - 1차 캐시와 동일성 보장
      - 같은 트랜잭션 안에서는 같은 엔티티르 ㄹ반환 - 약간의 조회 성능 향상
      - DB Isolation Level이 Read Commit이어도 애플리케이션에서 Repeatable Read 보장(SQL이 한번만 실행된다)
    - 트랜잭션을 지원하는 쓰기 지원
      - 데이터를 버퍼로 모은다. 트랜잭션을 커밋할 때까지 insert sql을 모은다.
      - jdbc batch sql 기능을사용해서 한번에 sql 전송(네트워크 비용 절약)
    - 지연로딩과 즉시 로딩(jpa는 둘다 지원한다)
      - 지연 로딩: 객체가 실제 사용될 때 로딩
      - 즉시 로딩: JOIN SQL로 한번에 연관된 객체까지 미리 조회
      - jpa 개발할 때 지연로딩으로 쭉 했다가 성능최적화 하면서 필요한 부분을 즉시로딩으로 교체

- ORM은 객체와 rdb 두 기중위에 있는 기술
- 관계형 db와 객체지향 둘 다 알고 고민해야 잘 쓸 수 있다.
