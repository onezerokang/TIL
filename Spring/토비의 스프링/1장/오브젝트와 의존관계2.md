# 1장: 오브젝트와 의존관계

스프링은 객체지향 프로그래밍을 지원하는 자바를 기반으로 한 기술이다. 스프링의 핵심 철학은 객체지향 프로그래밍이 제공하는 이점을 누릴 수 있도록 하는 것이다. 그래서 스프링을 이해하려면 오브젝트의 생성주기, 기술적인 특징, 사용방법에 대해 관심을 가져야 하며, 이러한 관심은 결국 오브젝트 설계로 발전한다.

객체지향 설계 원칙, 디자인 패턴, 리팩토링, 단위 테스트와 같은 오브젝트 설계와 구현에 관한 여러 가지 기술과 지식이 요구된다.

스프링은 객체지향 기술과 설계, 구현에 관한 베스트 프랙티스를 평범한 개발자도 자연스럽고 손쉽게 적용할 수 있도록 프레임워크 형태로 제공한다.

1장에서는 스프링이 관심을 갖는 대상인 오브젝트의 설계와 구현, 동작원리에 더 집중하기를 바란다. 그러다 보면 자연스럽게 스프링이 무엇인지도 이해될 것이다.

## 초난감 DAO

사용자 정보를 JDBC API를 통해 DB에 저장하고 조회할 수 있는 간단한 DAO를 하나 만들어보자.

### User

먼저 사용자 정보를 저장할 User 클래스를 만들자.

```java
public class User {
    String id;
    String name;
    String password;

    public String getId() {
        return id;
    }

    public void setId(String id) {
        this.id = id;
    }

    public String getName() {
        return name;
    }

    public void setName(String name) {
        this.name = name;
    }

    public String getPassword() {
        return password;
    }

    public void setPassword(String password) {
        this.password = password;
    }
}
```

User 오브젝트에 담긴 정보가 실제로 보관될 DB의 테이블을 하나 만들자. 필자는 MySQL을 사용했다.

```sql
create table users (
    id varchar(10) primary key,
    name varchar(20) not null,
    password varchar(10) not null);
```

### UserDao

사용자 정보를 DB에 넣고 관리할 수 있는 DAO 클래스를 만들자. 우선 생성(add)과 조회(get) 두 개의 메소드를 먼저 만들어보자.

```java
public class UserDao {
    public void add(User user) throws ClassNotFoundException, SQLException {
//        Class.forName("com.mysql.jdbc.Driver");
        Connection c = DriverManager.getConnection("jdbc:mysql://localhost:3306/toby_spring", "root", "1234");

        PreparedStatement ps = c.prepareStatement(
                "insert into users(id, name, password) values(?,?,?)");
        ps.setString(1, user.getId());
        ps.setString(2, user.getName());
        ps.setString(3, user.getPassword());

        ps.executeUpdate();

        ps.close();
        c.close();
    }

    public User get(String id) throws ClassNotFoundException, SQLException {
//        Class.forName("com.mysql.jdbc.Driver");
        Connection c = DriverManager.getConnection("jdbc:mysql://localhost:3306/toby_spring", "root", "1234");

        PreparedStatement ps = c.prepareStatement(
                "select * from users where id = ?");
        ps.setString(1, id);

        ResultSet rs = ps.executeQuery();
        rs.next();
        User user = new User();
        user.setId(rs.getString("id"));
        user.setName(rs.getString("name"));
        user.setPassword(rs.getString("password"));

        rs.close();
        ps.close();
        c.close();

        return user;
    }
}
```

- JDBC 4.0 이상부터는 Class.forName()으로 드라이버를 로드하지 않아도 된다. DriverManager.getConnection()이 호출될 때 자동으로 드라이버가 로드된다.

- Docker 환경에서 MySQL 로그인 시 Access denied for user 'mysql'@'172.17.0.1' (using password: YES) 문제

### main()을 이용한 DAO 테스트 코드

위에서 만든 UserDao가 제대로 동작하는지 확인하기 위해서 오브젝트 스스로가 자신을 검증하도록 해보자.

방법은 다음과 같다.

1. UserDao에 main() 메소드를 추가한다.
2. User 오브젝트를 생성하고 프로퍼티에 값을 넣은 다음 add() 메소드를 이용해 DB에 등록해보자.
3. get() 메소드를 사용해 저장된 결과를 확인하자.

```java
public static void main(String[] args) throws SQLException, ClassNotFoundException {
    UserDao dao = new UserDao();

    User user = new User();
    user.setId("faker");
    user.setName("이상혁");
    user.setPassword("1234");

    dao.add(user);

    System.out.println(user.getId() + " 등록 성공");

    User user2 = dao.get(user.getId());
    System.out.println(user2.getName());
    System.out.println(user2.getPassword());

    System.out.println(user2.getId() + " 조회 성공");
}
```

코드가 정상동작 하였다면 아래와 같은 메시지가 출력될 것이다.

```
faker 등록 성공
이상혁
1234
faker 조회 성공
```

이렇게 초간단 DAO와 테스트용 메소드까지 만들었다.

하지만 이 DAO코드는 실무에서 이렇게 하면 초난감해질 조건을 두루 갖춘 DAO 코드다.

지금부터 초난감 DAO 코드를 객체지향 기술의 원리에 충실한 멋진 스프링 스타일의 코드로 개선해볼 것이다.

## DAO의 분리

### 관심사의 분리

요구사항이 끊임없이 변함에 따라서 오브젝트의 설계와 이를 구현한 코드 또한 끊임없이 변경된다.

그래서 객체를 설계할 때 미래의 변화에 대한 대비를 염두해야 한다.

객체지향 기술은 가상의 추상세계를 효과적으로 구성할 수 있고, 이를 자유롭고 편리하게 변경, 발전, 확장시킬 수 있다.

변경이 일어날 때 필요한 작업을 최소화하고, 그 변경이 다른 곳에 문제를 일으키지 않게 하려면 **분리와 확장**을 구려한 설계를 해야 한다.

프로그래밍의 기초 개념 중에 **관심사의 분리(Separation of Concerns)**라는 게 있다. 이를 객체지향에 적용해보면 관심이같은 것끼리는 하나의 객체 안으로 또는 친한 객체로 모이게 하고, 관심이 다른 것은 가능한 따로 떨어져서 서로 영향을 주지 않도록 분리해야 한다.

### 커넥션 만들기의 추출

UserDao의 add() 메소드에는 적어도 세 가지 관심사항을 발견할 수 있다.

#### UserDao의 관심사항

1. DB 연결을 위한 커넥션 가져오기
2. SQL을 보내기 위해 Statement를 만들고 실행
3. 작업이 끝나면 Statement와 Connection 닫기

심지어 Connection 오브젝트를 가져오는 코드는 add()와 get() 메소드에 중복되어 있다. 현재는 메소드가 두 개뿐이지만 메소드가 더 증가하게 되면 중복은 더 늘어나게 될 것이고, 유지보수는 더욱 어려워질 것이다.

#### 중복 코드의 추출

getConnection() 메소드를 만들어 중복된 DB 연결 코드를 분리해주자.

```java
public void add(User user) throws SQLException {
    Connection c = getConnection();
    // ... 생략
}

public User get(String id) throws SQLException {
    Connection c = getConnection();
    // ... 생략
}

private Connection getConnection() throws SQLException {
    Connection c = DriverManager.getConnection("jdbc:mysql://localhost:3306/toby_spring", "root", "1234");
    return c;
}
```

이렇게 관심의 종류에 따라 코드를 구분해놓았기 때문에 한 가지 관심에 대한 변경이 일어날 경우 그 관심이 집중되는 코드만 수정하면 된다.

#### 변경사항에 대한 검증: 리팩토링과 테스트

이렇게 리팩토링을 한 후에는 반드시 검증을 해야 한다. 이전에 만들었던 main() 메소드를 호출하여 제대로 동작하는지 확인하자.

main() 메소드를 다시 호출한다면 에러 메시지가 출력될텐데 기본키인 id가 중복되기 때문이다. 이전에 생성한 레코드를 제거해주는 로직을 추가해줘도 되지만 일단 delete from users로 레코드를 제거하고 테스트해주자.

### DB 커넥션 만들기의 독립

위에서는 메소드 추출을 통해 변화에 유연하게 대처할 수 있는 코드를 작성했다. 이번에는 더 나아가서 변화에 대응하는 수준이 아닌 아예 변화를 반기는 DAO를 만들어보자.

다음은 요구사항이다.

1. UserDao가 발전에 발전을 거듭해서 전세계적으로 인기가 많아졌다.
2. 네이버와 다음이 UserDao를 구매하겠다는 주문이 들어왔다.
3. 문제는 네이버와 다음이 다른 종류의 DB를 사용하고 있고, DB 커넥션을 가져오는 데 있어 독자적으로 만든 방법을 적용하고 싶어한다.
4. UserDao의 소스 코드를 고객에게 제공하고 수정하라고 해도 되지만, 소스 코드를 직접 공개하고 싶지 않다. 고객에게는 미리 컴팡리된 클래스 바이너리 파일만 제공하고 싶다.
5. 이런 경우 UserDao 소스코드를 제공하지 않고 고객 스스로 원하는 DB 커넥션 생성 방식을 적용해가면서 UserDao를 사용하게 할 수 있을까?

#### 상속을 통한 확장

UserDao에서 메소드의 구현 코드를 제거하고 getConnection()을 추상 메소드로 만든다.

네이버와 다음은 추상 클래스가 되어버린 UserDao를 확장한 후 getConnection()만 오버라이딩할 수 있다.

기존에는 같은 클래스에서 다른 메소드로 분리됐던 DB 커넥션 연결이라는 관심을 이번에는 상속을 통해 서브 클래스로 분리해버리는 것이다.

```java
public abstract class UserDao {
    public void add(User user) throws SQLException {
        Connection c = getConnection();
        // ... 생략
    }

    public User get(String id) throws SQLException {
        Connection c = getConnection();
        // ... 생략
    }

    public abstract Connection getConnection() throws SQLException;
}
```

```java
public class NUserDao extends UserDao{
    @Override
    public Connection getConnection() throws SQLException {
        // 네이버의 connection 생성 코드
    }
}

public class DUserDao extends UserDao{
    @Override
    public Connection getConnection() throws SQLException {
        // 다음의 connection 생성 코드
    }
}
```

수정된 코드는 DAO의 핵심 기능인 데이터를 등록하고 가져오는 관심을 담당하는 UserDao와 DB 연결 방법에 대한 관심을 담고 있는 NUserDao와 DUserDao가 클래스 레벨로 구분 되고 있다. 클래스 계층구조를 통해 두 개의 관심이 독립적으로 분리되면서 변경 작업은 한층 용이해졌다.

이렇게 슈퍼클래스에 기본적인 로직의 흐름을 만들고, 그 기능의 일부를 추상 메소드나 오버라이딩이 가능한 protected 메소드 등으로 만든 뒤 서브 클래스에서 이런 메소드를 필요에 맞게 구현해서 사용하는 방법을 **템플릿 메소드 패턴**이라고 한다.

UserDao의 getConnection() 메소드는 Connection 타입 오브젝트를 생성한다는 기능을 정의해놓은 추상 메소드다. 그리고 UsreDao의 서브클래스인 getConnection() 메소드는 어떤 Connection 클래스의 오브젝트를 어떻게 생성할 것인지를 결정하는 방법이라고 볼 수 있다. 이렇게 서브 클래스에서 구체적인 오브젝트 생성 방법을 결정하게 하는 것을 **팩토리 메소드 패턴**이라고 부르기도 한다.

템플릿 메소드 패턴과 팩토리 메소드 패턴은 상속을 사용한다.

상속은 다음과 같은 단점을 갖는다.

1. 다중 상속을 허용하지 않는다: 상속으로 분리하면 나중에 다른 목적으로 상속을 적용하기 힘들다.
2. 상하위 클래스의 관계는 밀접하다: 상속관계는 두 가지 다른 관심사에 대해 긴밀한 결합을 허용한다. 슈퍼클래스 내부의 변경이 있을 때 모든 서브 클래스를 수정하거나 다시 개발해야 할 수 있다.
3. 확장된 DB 커넥션을 생성하는 코드를 다른 DAO에 적용할 수 없다는 것도 단점이다. DAO 클래스가 여러개라면 getConnection() 구현 코드가 매 DAO마다 중복될 것이다.

## DAO의 확장

추상 클래스를 만들고 이를 상속한 서브클래스에서 변화가 필요한 부분을 바꿔서 쓸 수 있게 만든 이유는 관심사가 다르고, 관심사가 다르기 때문에 변화의 성격이 다른 것을 분리해서, 서로 영향을 주지 않은 채로 각각 필요한 시점에 독립적으로 변경할 수 있게 하기 위해서다. 그러나 여러가지 단점이 많은 상속이라는 방법을 사용했다는 것이 불편하게 느껴진다.

### 클래스의 분리

이번에는 상속관계도 아닌 완전히 독립적인 클래스로 분리해보자. DB 커넥션과 관련된 부분을 서브클래스가 아닌 별도의 클래스로 담고 이를 UserDao가 사용하는 것이다.

```java
public class UserDao {

    private SimpleConnectionMaker simpleConnectionMaker;

    public UserDao() {
        simpleConnectionMaker = new SimpleConnectionMaker();
    }

    public void add(User user) throws SQLException {
        Connection c = simpleConnectionMaker.makeNewConnection();
        // ... 생략
    }

    public User get(String id) throws SQLException {
        Connection c = simpleConnectionMaker.makeNewConnection();
        // ... 생략
    }

    public static void main(String[] args) throws SQLException {
        // ... 검증 코드
    }
}


```

```java
public class SimpleConnectionMaker {

    public Connection makeNewConnection() throws SQLException {
        Connection c = DriverManager.getConnection("jdbc:mysql://localhost/toby_spring", "root", "1234");
        return c;
    }
}
```

이렇게 리팩토링을 한 후에는 꼭 main() 메소드를 실행해서 전과 동일한 결과가 나오는지 확인하자.

성격이 다른 코드를 독립적인 클래스로 잘 분리한 것 같기는 한데, 네이버와 다음에게 UserDao만 공급하고 상속을 통해 DB 커넥션 기능을 확장해서 사용했던 게 다시 불가능해졌다. 왜냐하면 UserDao 코드가 SimpleConnectionMaker라는 특정 클래스에 종속되어 있기 때문에 DB 커넥션 생성 기능을 변경할 방법이 없다.

## 인터페이스의 도입

두 개의 클래스 사이에 추상적인 느슨한 연결고리를 두어 결합도를 낮추어 이런 문제를 해결할 수 있다.

추상화란 어떤 것들의 공통적인 성격을 뽑아내어 이를 따로 분리해내는 작업이다.

자바가 추상화를 위해 제공하는 가장 유연한 도구는 바로 인터페이스다. 인터페이스는 구현 클래스에 대한 구체적인 정보는 모두 감춰버린다.

UserDao가 인터페이스를 사용한다면 인터페이스가 제공하는 기능만 알면 되지, 그 기능을 어떻게 구현했는지에는 관심을 둘 필요가 없다.

ConnectionMaker 인터페이스를 정의하자.

```java
public interface ConnectionMaker {
    public Connection makeConnection() throws SQLException;
}
```

고객에게 납품 시 UserDao 클래스와 함께 ConnectionMaker 인터페이스를 전달한다. 고객은 ConnectionMaker를 구현하여 DB 연결 방법을 결정할 수 있다.

```java
public class DConnectionMaker implements ConnectionMaker {
    @Override
    public Connection makeConnection() throws SQLException {
        return null;
    }
}
```

```java
public class UserDao {

    private SimpleConnectionMaker simpleConnectionMaker;

    public UserDao() {
        simpleConnectionMaker = new DConnectionMaker();
    }
```

## 관계설정 책임의 분리

인터페이스를 통해 UserDao와 ConnectionMaker를 분리했는데, 왜 여전히 UserDao가 구체적인 클래스까지 알아야 할까? 그 이유는 UserDao에 어떤 ConnectionMaker 구현 클래스를 사용할지를 결정하는 관심사항이 존재하기 때문이다. 이 관심사를 담은 코드를 UserDao에서 분리하지 않으면 UserDao는 결코 독립적으로 확장 가능한 클래스가 될 수 없다.

**내 생각: 의존성 주입을 해야 하는 이유는 구현체를 선택해야 하는 책임을 분리하기 위해서.**
**현재 UserDao는 ConnectionMaker와 DConnectionMaker 모두에 의존하고 있다**

UserDao를 사용하는 클라이언트 오브젝트가 UserDao와 ConnectionMaker의 구현 클래스의 관계를 결정해주는 기능을 분리해서 두기에 적절하다.
