# IoC와 DI

## 개요

- IoC와 DI는 스프링의 3대 구성요소 중 하나라고 불릴만큼 중요하다.
- 스프링 프레임워크는 IoC와 DI를 통해 결합도를 낮춰 유연하고 유지보수하기 쉬운 코드를 작성하게 도와준다.

## IoC(Inversion of Control, 제어의 역전)

- 제어의 역전은 프로그램 제어의 흐름이 뒤바뀌는 것이다.
- 일반적인 프로그램은 main() 메소드와 같이 프로그램 시작 지점에서 사용할 오브젝트를 결정하고, 생성하고, 메 소드를 호출하고, 다른 오브젝트를 의존한다. 즉 프로그램의 흐름과 객체의 제어권을 개발자가 작성한 코드를 갖는다.
- 제어의 역전은 이런 제어의 흐름을 뒤집는다. 오브젝트는 자신이 사용할 오브젝트를 스스로 선택하지도, 생성하지도 않는다. 자신이 어떻게 만들어지고 어디서 사용되는지 알 수 없다. 모든 제어 권한을 IoC Container에게 위임한다.
- IoC(Inversion of Control, 제어의 역전)은 객체의 생명주기를 개발자가 작성한 코드가 아닌 IoC Container가 갖는 것이다.
- 과거 프로그래밍 방식에서는 개발자가 클래스를 통해 객체의 구조를 정의하고, 객체를 생성하고 객체가 의존해야 하는 다른 객체 또한 직접 생성해줬다.
- 프로그램의 제어권을 개발자가 작성한 코드가 갖는 구조였다.
- 이런 구조의 문제는 객체가 객체를 의존할 때, 구현체를 직접 의존하기에 기능을 확장하거나 구현체를 변경하는데 어려움이 있었다.
- 따라서 객체를 생성하고, 의존관계를 주입해주는 책임을 객체가 아닌 IoC Container에게 위임하여 객체의 결합도를 낮춤
- 또한 프로그래머가 프로그램이 동작하기 위한 것들을 모두 구현하지 않고, IoC Container에 위임하여 비즈니스 로직에 집중할 수 있게 한다.

## DI(Dependency Injection, 의존성 주입)

- DI(Dependency Injection, 의존성 주입)는 의존해야 하는 객체를, 클라이언트 객체 내부에서 직접 생성하고 사용하는 것이 아니라 외부에서 IoC Container가 생성을 하고, 주입
- 객체를 직접 의존하는 것이 아니라 추상 계층인 인터페이스를 통해 의존하기에 구현 객체를 변경해도 그대로 동작한다.

## 애플리케이션 컨텍스트

- 스프링에서 IoC Container를 빈 팩토리(bean factory)라고 한다. 보통은 빈 팩토리를 조금 더 확장한 어플리케이션 컨텍스트(application context)를 주로 사용한다.
- 어플리케이션 컨텍스트는 설정 정보를 참고해서 빈을 생성하고, 관계설정 등의 제어 작업을 총괄한다.
- 싱글톤 레지스트리 오브젝트 스코프
  - 애플리케이션 컨텍스트에서 여러번 빈을 요청할 경우 매번 동일한 빈을 돌려준다.
  - 애플리케이션은 IoC 컨테이너면서 동시에 싱글톤을 저장하고 관리하는 싱글톤 레지스트리다.
  - 하지만 디자인 패턴에 나오는 싱글톤 패턴과 비슷한 개념이지만 구현 방법은 확연히 다르다.
  - 요청이 들어올 때마다 매번 객체를 새로 만들면 서버가 감당학 ㅣ힘들다.
- 싱글톤 패턴의 문제점을 보완하면서 싱글톤 방식으로 객체를 관리할 수 있다.
