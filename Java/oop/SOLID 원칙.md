# SOLID 원칙

## 1. SOLID 원칙이란

SOLID 원칙은 로버트 C. 마틴이 발표한 5가지 객체지향 설계 원칙의 약자다. 각 글자는 다음과 같은 원칙의 약자다.

- SRP(Single Responsibility Principle): 단일 책임 원칙
- OCP(Open Closed Principle): 개방 폐쇄 원칙
- LSP(Liskov Substitution Principle): 리스코프 치환 원칙
- ISP(Interface Segregation Principle): 인터페이스 분리 원칙
- DIP(Dependency Inversion Principle): 의존 역전 원칙

이 원칙들의 목표는 응집도는 높이고, 결합도를 낮추자는 고전 원칙을 객체지향의 관점에서 재정의 한 것이다.

## 3. SRP - 단일 책임 원칙

> 어떤 클래스를 변경해야 하는 이유는 오직 하나뿐이어야 한다
>
> - 로버트 C. 마틴

단일 책임 원칙은 하나의 클래스는 하나의 책임만을 가져야 한다는 원칙이다. 만약 하나의 클래스에 다수의 역할과 책임이 몰려있을 경우 클래스를 역할과 책임에 따라 쪼개 각각 하나의 역할과 책임만 갖게 해야 한다. 단일 책임 원칙은 클래스 뿐만 아니라 메소드, 패키지 등에도 적용될 수 있다.

왜 하나의 책임만을 가져야 할까?

1. 코드가 지저분해진다.
2. 하나의 책임만 수정해야 하는 상황에서, 실수로 다른 책임까지 건드리게 되어 예상하지 못한 버그를 발생시킬 수 있다.

## 4. OCP - 개방 폐쇄 원칙

> 소프트웨어 엔티티(클래스, 모듈, 함수 등)는 확장에 대해서는 열려 있어야 하지만 변경에 대해서는 닫혀 있어야 한다.
>
> - 로버트 C. 마틴

걔방 폐쇄 원칙은 소프트웨어 컴포넌트는 확장에는 열려 있어야 하고, 수정에 관해서는 닫혀 있어야 한다는 의미다.

보통 의존관계 역전 원칙과 밀접한 원칙이 OCP다. 구체가 아니라 추상에 의존하면 확장에는 열려있고, 수정에는 닫혀 있는 코드를 작성할 수 있기 때문이다.

## 5. LSP - 리스코프 치환 원칙

> 서브 타입은 언제나 자신의 타입 기반으로 교체할 수 있어야 한다.
>
> - 로버트 C. 마틴

리스코프 치환 원칙은 파생 타입은 반드시 기본 타입을 완벽하게 대체할 수 있어야 한다는 사실이다.

리스코프 치환 원칙은 자바의 다형성을 보여주는 원칙이다.

- 하위 클래스 is a kind of 상위 클래스 - 하위 분류는 상위 분류의 한 종류이다.
- 구현 클래스 is able to 인터페이스 - 구현 분류는 인터페이스할 수 있어야 한다.

위 두 문장을 지키고 있다면 이미 리스코프 치환 원칙을 잘 지키고 있는 것이다.

위 문장대로 구현되지 않은 코드는 상속이 조직도나 계층도 형태로 구축된 경우다.

```java
// 딸은 아빠의 일종이다(문장이 성립하지 않는다).
// 즉 하위 클래스가 상위 클래스의 인스턴스 역할을 할 수 없다.
아버지 춘향이 = new 딸();
```

하위 클래스의 인스턴스는 상위형 객체 참조 변수에 대입해 상위 클래스의 인스턴스 역할을 하는 데 문제가 없어야 한다.

## 6. ISP - 인터페이스 분리 원칙

> 클라이언트는 자신이 사용하지 않는 메소드에 관계를 맺으면 안 된다.
>
> - 로버트 C. 마틴

인터페이스 분리 원칙은 클라이언트가 사용하지 않을 불필요한 메소드를 강제로 구현하게 해서는 안된다는 원칙이다. 클라이언트가 사용하지 않을 메소드를 강제로 구현하는 일이 없을 때까지 하나의 인터페이스를 2개의 인터페이스로 분할하는 원칙이다. 인터페이스가 최소한의 책임만 가져야 한다.

## 7. DIP - 의존관계 역전 원칙

> 고차원 모듈은 저차원 모듈에 의존하면 안 된다. 이 두 모듈 모두 다른 추상화된 것에 의존해야 한다.추상화된 것은 구체적인 것에 의존하면 안 된다. 구체적인 것이 추상화된 것에 의존해야 한다. 자주 변경되는 구체 클래스에 의존하지 마라
>
> - 로버트 C. 마틴

의존관계 역전 원칙은 구체화가 아닌 추상화에 의존해야 한다는 의미다. 어떤 객체가 다른 객체에 직접적으로 의존하는 것이 아닌, 구상 모듈을 결합하기 위한 추상 계층을 사용한다는 것을 의미한다.

그렇다면 왜 구체화가 아닌 추상화에 의존해야 할까? 그 이유는 구체화가 추상화보다 변경 가능성이 높기 떄문이다. 변경 가능성에 높은 코드를 의존하게 되면 클라이언트 객체도 함께 변경되어야 한다. 하지만 변경이 잘 발생하지 않는 추상화에 의존하게 되면 구체화가 변경되어도 클라이언트 객체는 변경되지 않는다.

이렇게 하면 분리된 상태를 유지하면서 다른 구성 모듈의 기능을 확장할 수 있다. 결합도를 낮춘다. 의존관계 역전 원칙을 따르면 개방 폐쇄 원칙도 지킬 수 있다.

## 8. 정리

SOLID 원칙은 결합도는 낮추고 응집도를 높이기 위한 객체지향 설계 원칙이다. 이 설계 원칙을 위반하며 프로그램을 짜면 유지보수하기 힘든 프로그램이 될 것이다. 클래스를 작성할 때 SOLID 원칙을 잘 지키고 있는지 주의하며 프로그래밍하자.
