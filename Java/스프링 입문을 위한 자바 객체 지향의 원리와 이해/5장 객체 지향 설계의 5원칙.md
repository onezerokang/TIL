# 5장 객체 지향 설계의 5원칙

## 1. 개요

객체 지향 설계의 5가지 원칙은 앞글자를 따서 SOLID 원칙이라고도 한다. SOLID 원칙은 로버트 C. 마틴이 2000년대 초반 객체 지향 프로그래밍 및 설계의 다섯 가지 기본 원칙으로 제시한 것을 마이클 페더스가 두문자어로 소개한 것이다.

다음은 SOLID 원칙이다.

- SRP(Single Responsibility Principle): 단일 책임 원칙
- OCP(Open Closed Principle): 개방 폐쇄 원칙
- LSP(Liskov Substitution Principle): 리스코프 치환 원칙
- ISP(Interface Segregation Principle): 인터페이스 분리 원칙
- DIP(Dependency Inversion Principle): 의존 역전 원칙

이 원칙들은 응집도는 높이고, 결합도를 낮추자는 고전 원칙을 객체 지향의 관점에서 재정의 한 것이다.

## 2. 응집도와 결합도

유지 보수하기 용이한 프로그램을 만들기 위해서는 응집도는 높이고, 결합도를 낮춰야 한다. 그렇다면 응집도와 결합도가 무엇일까?

- **응집도(Cohesion)**: 한 모듈안에 해당 모듈이 갖고 있는 책임과 연관된 요소로 이루어져있는 것. 어떤 기능을 수정할 때 해당 기능이 관련 있는 것끼리 잘 모듈화 되어있기 때문에 해당 부분만 수정하면 된다.
- **결합도(Coupling)**: 다른 모듈과의 의존성 정도. 결합도가 높으면 하나의 모듈을 수정할 때 결합되어 있는 다른 모듈도 수정해야 한다.

## 3. SRP - 단일 책임 원칙

> 어떤 클래스를 변경해야 하는 이유는 오직 하나뿐이어야 한다
>
> - 로버트 C. 마틴

하나의 클래스는 하나의 책임을 가져야 한다는 원칙이다. 만약 하나의 클래스에 다수의 역할과 책임이 몰려 있을 경우 클래스를 역할과 책임에 따라 분리하여 각각 하나의 역할과 책임만 갖게 해야 한다. 단일 책임 원칙은 클래스 뿐만 아니라 속성, 메소드, 패키지, 모듈, 컴포넌트, 프레임워크 등에도 적용할 수 있다.

### 3.1. 예제1 - 속성이 단일 책임 원칙을 지키지 않은 경우

아래의 '사람' 클래스는 '군번'이라는 속성을 갖는다. 문제는 '군번'은 남자만 가져야 할 속성인데 여자 객체의 군번 속성에 값을 할당하거나 읽어오는 코드를 제제할 방법이 없다.

```java
class 사람 {
	String 군번;
}

사람 로미오 = new 사람();
사람 줄리엣 = new 사람();

로미오.군번 = "1234567"; // 문제 없음
줄리엣.군번 = “1234567”; // 군번 속성을 갖으면 안됨
```

사람 클래스가 남자의 역할과 여자의 역할을 전부 하게 하지 말고 단일 책임 원칙을 적용하자. 남자 클래스와 여자 클래스가 공통점이 많다면 사람 클래스를 상위 클래스로 하고, 공통점이 없다면 사람 클래스를 제거하면 된다.

- **공통점이 많은 경우**

```java
abstract class 사람 {
	// 공통으로 가져야 할 추상 클래스
}

class 남자 extends 사람 {
	String 군번;
}

class 여자 extends 사람 {}
```

- **공통점이 없는 경우**

```java
class 남자 {
	String 군번;
}

class 여자 {}
```

이 외에도 하나의 속성이 여러 의미를 갖는 경우도 단일 책임 원칙을 지키지 못한 것이다.

### 3.2. 예제2 - 메서드가 단일 책임 원칙을 지키지 못하는 경우

아래 메소드는 수컷 강아지의 행동과 암컷 강아지의 행동을 모두 구현하려 하기에 단일 책임 원칙을 위배한다.

```java
class 강아지 {
	final static Boolean 수컷 = true;
	final static Boolean 암컷 = false;
	Boolean 성별;

	void 소변보다() {
		if (this.성별 == 수컷) {
			// 한쪽 다리를 들고 소변을 본다.
		} else {
			// 뒷다리 두 개를 굽혀 앉은 자세로 소변을 본다.
		}
	}
}
```

메서드가 단일 책임 원칙을 지키지 않은 경우, 분기 처리를 위한 if이 자주 보이게 된다.

위 코드를 수컷강아지 클래스와 암컷 강아지 클래스로 분리해보면 아래와 같이 된다.

```java
abstarct class 강아지 {
	abstract void 소변보다()
}

class 수컷강아지 extends 강아지 {
	void 소변보다() {
		// 한쪽 다리를 들고 소변을 본다.
	}
}

class 암컷강아지 extends 강아지 {
	void 소변보다() {
		// 뒷다리 두 개를 굽혀 앉은 자세로 소변을 본다.
	}
}
```

객체 지향 4대 특정 중 단일 책임 원칙과 가장 관계가 깊은 것은 모델링하는 과정을 담은 추상화다.
애플리케이션의 경계를 정하고 추상화를 통해 클래스들을 선별하고 속성과 메소드를 설계할 때 반드시 단일 책임 원칙을 고려하는 습관을 들이자.

## 4. OCP - 개방 폐쇄 원칙

> 소프트웨어 엔티티(클래스, 모듈, 함수 등)는 확장에 대해서는 열려 있어야 하지만 변경에 대해서는 닫혀 있어야 한다.
>
> - 로버트 C. 마틴

개방 폐쇄 원칙을 처음 접한 사람들은 '확장에는 열려 있지만 변경에는 닫혀 있다'는 말이 모순적이라고 느낄 수 있다. 이를 의역하면 '자신의 확장에는 열려 있으나, 주변의 변화에는 닫혀있다'고 할 수 있다.

개방 폐쇄 원칙을 잘 지킨 예시로 JDBC가 있다. 자바로 데이터베이스 프로그래밍을 할 때 자바 어플리케이션은 JDBC라는 인터페이스에 의존하기 떄문에 실제 데이터베이스가 변경되어도 자바 어플리케이션은 변경하지 않아도 된다(물론 Connection 설정은 변경해야 한다).

자바 어플리케이션은 데이터베이스 변화에는 닫혀 있지만, 데이터베이스는 자신의 확장에는 열려 있다.

개방 폐쇄 원칙을 지키지 않으면 객체 지향 프로그램의 장점인 유연성, 재사용성, 유지보수성 등을 얻을 수 없다. 따라서 개방 폐쇄 원칙은 반드시 지켜져야 한다.

## 5. LSP - 리스코프 치환 원칙

> 서브 타입은 언제나 자신의 타입 기반으로 교체할 수 있어야 한다.
>
> - 로버트 C. 마틴

객체 지향에서 상속은 조직도나 계층도가 아닌 분류도가 돼야 한다.
객체 지향의 상속은 다음 조건을 만족해야 한다.

- 하위 클래스 is a kind of 상위 클래스 - 하위 분류는 상위 분류의 한 종류이다.
- 구현 클래스 is able to 인터페이스 - 구현 분류는 인터페이스할 수 있어야 한다.

위 문장대로 구현되지 않은 코즈는 상속이 조직도나 계층도 형태로 구축된 경우다.

아버지 춘향이 = new 딸(); // 딸에게 아빠의 역할을 맞추고 있다.

하위 클래스의 인스턴스는 상위형 객체 참조 변수에 대입해 상위 클래스의 인스턴스 역할을 하는 데 문제가 없어야 한다.

## 6. ISP - 인터페이스 분리 원칙

> 클라이언트는 자신이 사용하지 않는 메소드에 관계를 맺으면 안 된다.
>
> - 로버트 C. 마틴

단일 책임 원칙과 인터페이스 분리 원칙은 같은 문제에 대한 두 가지 다른 해결책이라고 볼 수 있다.
프로젝트 요구사항과 설계자의 취향에 따라 단일 책임 원칙이나 인터페이스 분할 원칙 중 하나를 선택해서 설계할 수 있다.
하지만 특별한 경우가 아니라면 단일 책임 원칙을 적용하는 것이 더 좋은 해결책이라고 할 수 있다.

인터페이스 분할 원칙을 이야기 할 때 인터페이스 최소 주의 원칙이 항상 언급된다.
인터페이스를 통해 외부에 메소드를 제공할 때는 최소한의 메소드만 제공하라는 것이다.
그 이유에 대해 알아보자. 리스코프 치환 원칙에 따라 하위 객체는 상위 객체인 척 할 수 있다.
상위 클래스가 빈약할 경우 하위 클래스가 공통된 속성과 메소드를 갖게 되고 형 변환이 자주 발생한다.

```java
사람 김학생 = new 학생(“김학생”, new Date(2001, 01, 01), “20010101-1234567”);
사람 이군인 = new 학생(“이군인”, new Date(2001, 01, 01), “20010101-1234567”);

System.out.println(김학생.생일); // 사용 불가
System.out.println(이군인.생일); // 사용 불가

System.out.println((학생)김학생.생일); // 캐스팅 필요
System.out.println((군인)이군인.생일); // 캐스팅 필요
```

물론 김학생 객체 참조 변수를 사람형이 아닌 학생형으로 만들면 되겠지만 그럼 굳이 상속 구조를 만들 필요도 없다.
상위 클래스형의 참조 변수를 이용해야 상속의 가장 큰 혜택을 볼 수 있다.

## 7. DIP - 의존 역전 원칙

> 고차원 모듈은 저차원 모듈에 의존하면 안 된다. 이 두 모듈 모두 다른 추상화된 것에 의존해야 한다.추상화된 것은 구체적인 것에 의존하면 안 된다. 구체적인 것이 추상화된 것에 의존해야 한다. 자주 변경되는 구체 클래스에 의존하지 마라
>
> - 로버트 C. 마틴

자동차는 몇 년은 타야 하는데 스노우타이어는 계절이 바뀌면 일반 타이어로 교체 해야 한다.
그런데 자동차가 자신보다 자주 변경되는 스노우타이어에 의존할 경우, 타이어를 교체할 때 부서지기 쉬움이라는 나쁜 냄새를 풍기게 된다.

자동차가 구체적인 타이어들이 아닌 추상화된 타이어 인터페이스에만 의존하게 함으로써 다른 타이어로 변경돼도 자동차는 이제 그 영향을 받지 않는 형태로 구성된다. OCP 설명할 때 들었던 것이다. 하나의 해결책을 찾으면 그 안에 여러 설계 원칙이 녹아 있는 경우가 많다.

의존 역전 원칙: 자신보다 변하기 쉬운 것에 의존하던 것을 추상화된 인터페이스나 상위 클래스를 두어 변하기 쉬운 것의 변화에 영향받지 않게 하는 것.
의역: 자신보다 변하기 쉬운 것에 의존하지 마라.

상위 클래스일수록, 인터페이스일수록, 추상 클래스일수록 변하지 않을 가능성이 높기에 이에 의존하는 것이다.
의존 역전 원칙을 적용한 사례는 JDBC에서도 확인할 수 있다.

## 8. 정리 - 객체 지향 설계와 SOLID

SOLID는 객체 지향을 올바르게 프로그램에 녹여내기 위한 원칙이다.
객체 지향 4대 원칙을 잘 이해해야 SOLID를 제대로 이해하고 활용할 수 있다.

SOLID를 이야기할 때 빼 놓을 수 없는 것이 SoC(Separation Of Concerns), 관심사의 분리다.
관심이 같은 것끼리 하나의 객체 안으로 또는 친한 객체로 모으고, 관심이 다른 것은 따로 떨어져 서로 영향을 주지 않도록 분리.
SoC를 적용하면 자연스럽게 단일 책임 원칙, 개방 폐쇄 원칙, 인터페이스 분리 원칙에 도달하게 된다.

- SRP(단일 책임 원칙): 어떤 클래스를 변경해야 하는 이유는 오직 하나뿐이어야 한다.
- OCP(개방 폐쇄 원칙): 자신의 확장에는 열려 있고, 주변의 변화에는 닫혀 있어야 한다.
- LSP(리스코프 치환 원칙): 서브 타입은 언제나 자신의 기반 타입으로 교체할 수 있어야 한다.
- ISP(인터페이스 분리 원칙): 클라이언트는 자신이 사용하지 않는 메소드에 의존 관계를 맺으면 안된다.
- DIP(의존 역전 원칙): 자신보다 변하기 쉬운 것에 의존하지 마라.
