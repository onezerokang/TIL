# 3장 객체 지향의 4대 특성

## 1. 객체 지향은 인간지향이다

객체 지향 프로그래밍은 기계에 맞춰 사고하는 방식을 버리고 현실 세계를 인지하는 방식으로 프로그램을 만들기에 직관적이다.

객체 지향 이전에는 속성과 메소드를 객체라는 단위로 묶지 않고 따로 분류된 형태로 프로그램을 작성했는데, 객체 지향에서는 우리가 실제 사물을 인지하고 사고하는 방식대로 객체 단위의 프로그래밍이 가능하다.

## 2. 클래스 vs 객체

객체 지향을 이해하는데 몇년이 걸린다고 말하는 사람들이 많다. 정확히 말하면 객체 지향에 대한 잘못된 이해를 바로 잡는 데 걸리는 시간이 몇 년이다. 잘못된 설명의 대표적인 예는 붕어빵틀(클래스)과 붕어빵(객체) 설명이다.

이는 클래스가 객체를 생성하는 것에만 포커스가 맞춰져 발생한 잘못된 설명이다. 해당 설명이 맞기 위해서는 다음 코드가 성립해야 한다.

```java
붕어빵틀 붕어빵 = new 붕어빵틀();
```

그렇다면 여기서 붕어빵틀을 생성하는 금형 기계가 있다면 금형 기계는 클래스고 붕어빵틀은 객체가 되버린다. 분명 붕어빵틀은 클래스라고 했는데 말이다.

```java
금형기계 붕어빵틀 = new 금형기계();
```

붕어빵틀은 붕어빵을 생성하는 팩터리라고 보는게 맞다.

클래스는 분류에 대한 개념이지 실체가 아니다.

## 3. 추상화: 모델링

### 3.1. 추상화는 모델링이다

추상화란 구체적인 것을 분해해서 관찰자가 관심 있는 특성만 가지고 재조합하는 것이다.

프로그래밍에서 추상화란 구체적인 것을 분해해서 애플리케이션 경계에 있는 특성만을 가지고 재조립하는 것이다. 그리고 이를 모델링이라고도 할 수 있다.

이런 추상화(모델링)은 자바에서 클래스, 인터페이스 등으로 구현할 수 있다.

### 3.3. 클래스 멤버 vs. 객체 멤버 = static 멤버 vs. 인스턴스 멤버

클래스는 분류이기에 속성을 가질 수 없고, 객체는 실체이기에 속성을 가질 수 있다. 하지만 모든 객체가 공통적으로 같은 값을 같는 속성이 있다면 static 키워드를 이용해서 한 곳에서 관리할 수 있다.

## 4. 상속: 재사용 + 확장

객체 지향의 상속 관계는 할아버지 -> 아버지 -> 자식 같은 조직도가 아니다.

동물 -> 포유류 -> 고래 같은 분포도라고 봐야 한다.

상속은 상위 클래스의 특성을 하위 클래스에서 상속하고 거기에 더해 필요한 특성을 추가하여 사용할 수 있다는 의미다.

### 4.1. 상속은 is a 관계를 만족해야 한다?

상속이 서브클래스 is a 슈퍼클래스 관계를 만족해야 한다고 하는 사람이 있는데 이는 오해다. is a kind of 관계를 만족해야 한다.

### 4.2. 다중 상속과 자바

자바는 다중 상속의 득보다 실이 크다고 판단하여 다중 상속을 지원하지 않는다.

```java
class A {
    hi() {
        System.out.println("하위");
    }
}

class B {
    hi() {
        System.out.println("헬롱");
    }
}

class C extends A, B {}
```

위 코드를 보면 C는 A와 B를 다중 상속하였다. 이때 A의 hi와 B의 hi중 무엇을 호출해야 할지 애매해지는 문제가 발생한다. 때문에 자바는 다중 상속을 포기하고 자바는 인터페이스를 도입해 다중 상속의 득은 취하고 실은 과감히 버렸다.

### 4.3. 상속과 인터페이스

- **상속**:
  - 상속 관계: 하위클래스 is a kind of 상위 클래스
  - 해석: 하위 클래스는 상위 클래스의 한 분류다.
  - 예제: 고래는 동물의 한 분류다.
- **인터페이스**:
  - 인터페이스: 구현 클래스 is able to 인터페이스
  - 해석: 구현 클래스는 인터페이스할 수 있다.
  - 예제: 고래는 헤엄칠 수 있다.

인터페이스는 "~할 수 있다"는 기능을 구현하도록 강조한다.

### 4.4 상속과 T 메모리

본 내용은 매우 중요하다.

상속 받은 클래스의 인스턴스를 생성하면 힙 영역에는 하위 클래스의 인스턴스뿐만 아니라 상위 클래스의 인스턴스도 생성된다(생성자 호출 시 super()를 호출하기 때문으로 보임).

이때 객체를 참조하는 변수는 생성된 실제 객체를 가리키는 것이 아닌 참조 타입의 객체를 가리킨다.

- 예시:

```java
Penguin pingu = new Penguin(); // pingu는 Penguin 인스턴스를 가리키지만
Animal pororo = new Penguin(); // pororo는 Animal 인스턴스를 가리킨다.

```

## 5. 다형성: 사용편의성

객체 지향에서 다형성이라고 하면 오버라이딩과 오버로딩이라고 할 수 있다.
물론 상위 클래스에서 하위 클래스 사이에서도 다형성을 이야기할 수 있고,
인터페이스와 그것의 구현 클래스 사이에서도 다형성을 이야기할 수 있지만
가장 기본은 오버라이딩과 오버로딩이라고 할 수 있다.

### 5.1. 다형성과 T 메모리

```java
Animal pororo = new Penguin(); // pororo는 Animal 인스턴스를 가리키지만
pingu.move(); // Penguin에서 Animal의 메소드를 오버라이딩했다면 오버라이딩한 메소드를 호출한다.
```

## 6. 캡슐화: 정보 은닉

자바에서 정보 은닉(information hiding)이라고 하면 접근 제어가가 생각날 것이다.
접근 제어자가 인스턴스 변수와 쓰일 때와 클래스 변수와 함께 쓰일 때를 비교해보자

### 6.1. 객체 멤버의 접근 제어자

자신의 멤버가 아닌 다른 객체의 멤버 변수에 접근할 때는 다른 객체를 생성한 후 접근해야 한다.

흔히 다음과 같은 그림(p.138로 접근제어자를 외우고 있다)

- public - 모두가 접근 가능
- protected - 삭송 / 같은 패키지 내의 클래스에서 접근 가능
- default - 같은 패키지 내에서만 가능
- private - 본인만 접근 가능

접근 제어자는 위와 같이 절대 단순하지 않다.
특히 객체 멤버에 대한 접근인가, 정적 멤버에 대한 접근인가에 따라 생각할 것이 매우 많아진다.

- 상속을 받지 않았다면 객체 멤버는 객체를 생성한 후 객체 참조 변수를 이용해 접근해야 한다.
- 정적 멤버는 클래스명.정적멤버 형식으로 접근하는 것을 권장한다.(일관된 형태로 접근하기 위해)
