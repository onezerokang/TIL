# 구현

- 구현 문제를 처음 접하면 생각나는 대로 코드를 작성하다 실수한다.
- 다음은 구현 문제를 접근하는 방법이다.

## 문제 나누어서 생각하기: 모듈화

- 코드를 한 번에 작성하지 않고, 역할별로 나누어 부분적으로 작성하면 한 번에 한 가지만 집중할 수 있다.
- 모듈화의 핵심은 코드를 역할별로 나누어 작성하는 것이다.
- 모듈화에서 가장 작은 단위는 메서드. 처음에 역할별로 메서드를 분리하여 작성해보자.
  - 메서드를 작성할 때는 메서드가 동작할 수 있는 최소한의 매개변수를 받도록 하자.
  - 최소한의 매개변수를 정의할 수 있다는 것은 메서드 역할을 확실히 정의했다는 의미다.
- 클래스로 분리도 좋은 방법
  - 데이터를 담는 클래스를 작성하고, 클래스의 메서드로 해당 클래스의 객체를 다루는 방법을 작성하면 부분적인 로직에 집중 가능
  - 클래스 또한 최소한의 데이터와 메서드를 포함하도록 작성한다.
- 문제가 어떻게 구성되어 있는지 확인하고, 문제 자체를 나누어서 생각해보자.

## 디버깅 빨리하기

- 하나의 역할을 하는 메서드나 클래스를 작성할 때마다 입력값을 넣고 코드를 실행해보자.

## 문제에서 이야기하는 대로 만들기

- 구현 문제는 규칙을 찾고 생각을 코드로 옮기는 문제
- 처음부터 최적화를 생각하지말고 요구 조건을 이해한 그대로 옮기는 것이 효율적이다.
- 처음부터 효율성을 고려하면 조건에 맞지 않는 코드를 짜거나, 시간이 부족할 수 있다.

## 규칙 찾아보기

- 로직을 생각할 때 메인 규칙과 예외 상황에 적용되는 규칙을 찾고, 이를 문제의 요구 조건에서 확인하여 처리한다.

## 완전 탐색 기반으로 풀기

- 완전 탐색에서는 상태 전이를 이용해서 탐색한다.

### 깊이 우선 탐색(DFS)

- 재귀를 이용하면 쉽게 구현할 수 있다.
- DFS는 탐색 공간이 제한되어 있고, 탐색 공간 내 탐색 목표가 있는지 검사할 때 유용하다.
- DFS는 재귀로 구현되므로 재귀의 특성을 갖는다. 따라서 현재 검사하는 상태에서 전이할 수 있는 상태가 있으면 해당 상태로 우선 전이한다.
- 따라서 탐색 공간의 깊이가 제한되어 있디 않은 문제에서 적용하기 힘들다. 무한히 진행될 수 있기 때문이다.
- 최단 경로를 찾지 못함.
- 위 단점이 있지만 BFS에 비해 작은 공간 복잡도를 갖는다는 장점이 있다.
