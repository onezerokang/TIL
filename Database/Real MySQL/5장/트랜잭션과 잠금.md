# 트랜잭션과 잠금

- **트랜잭션(Transaction)**: 일부만 실행되어서는 안되는 여러 쿼리들을 모두 실행되거나, 모두 실패하도록 하는 기능
- **잠금(Lock)**: 여러 커넥션에서 동시에 동일한 자원을 요청할 경우 순서대로 한 시점에는 하나의 커넥션만 변경할 수 있게 해주는 역할을 한다.
- **트랜잭션과 락의 차이**: 락과 트랜잭션은 서로 비슷한 개념 같지만 락은 동시성을 제어하기 위한 기능이고, 트랜잭션은 데이터의 정합성을 보장하기 위한 기능이다.
- **격리수준(Isolation level)**: 하나의 트랜잭션 내에서 또는 여러 트랜잭션 간의 작업 내용을 어떻게 공유하고 차단할 것인지를 결정하는 레벨

## 트랜잭션

이번절에서는 트랜잭션을 지원하지 않는 MyISAM과 트랜잭션을 지원하는 InnoDB의 처리 방식 차이를 살펴보겠다.

### MySQL에서의 트랜잭션

트랜잭션은 꼭 여러 개의 변경 작업을 수행하는 쿼리가 조합됐을 때만 의미 있는 개념은 아니다. 트랜잭션은 하나의 논리적인 작업셋에 쿼리가 몇개 있든 관계없이 100%으로 적용되거나, 아무것도 적용되지 않아야 한다는 것을 보장해 주는 것이다.

트랜잭션 관점에서 InnoDB 테이블과 MyISAM 테이블의 차이를 살펴보자.

```SQL
CREATE TABLE tab_myisam (fdpk INT NOT NULL, PRIMARY KEY(fdpk)) ENGINE=MyISAM;
INSERT INTO tab_myisam (fdpk) VALUES (3);

CREATE TABLE tab_innodb (fdpk INT NOT NULL, PRIMARY KEY(fdpk)) ENGINE=INNODB;
INSERT INTO tab_innodb (fdpk) VALUES (3);
```

위와 같이 테스트용 테이블에 각각 레코드를 1건씩 저장한 후 AUTO-COMMIT 모드에서 다음 쿼리문을 각각 실행해보자.

```SQL
SET autocommit=ON;

mysql> INSERT INTO tab_myisam (fdpk) VALUES (1), (2), (3);
ERROR 1062 (23000): Duplicate entry '3' for key 'tab_myisam.PRIMARY'

mysql> INSERT INTO tab_innodb (fdpk) VALUES (1), (2), (3);
ERROR 1062 (23000): Duplicate entry '3' for key 'tab_innodb.PRIMARY'

mysql> select * from tab_myisam;
+------+
| fdpk |
+------+
|    1 |
|    2 |
|    3 |
+------+
3 rows in set (0.00 sec)

mysql> select * from tab_innodb;
+------+
| fdpk |
+------+
|    3 |
+------+
```

두 테이블 모두 기존에 fdpk가 3인 레코드가 있었기에 키 중복 오류로 쿼리가 실패할 것이다. 그런데 각 테이블의 레코드를 조회해보면 MyISAM은 오류가 발생했음에도 '1'과 '2'는 INSERT 상태로 남아 있는 것을 확인할 수 있다.

부분 업데이트 문제 -> 정합성을 맞추는 데 상당히 어려운 문제를 만들어낸다.

### 주의사항

트랜잭션은 필요한 최소한의 코드에만 적용하는 것이 좋다. 이는 프로그램 코드에서 트랜잭션의 범위를 최소화하라는 의미다.

다음은 사용자가 게시판에 게시물을 작성한 후 저장 버튼을 클릭했을 때 서버에서 처리하는 내용을 순서대로 정리한 것이다.

```
1) 처리 시작
  => 커넥션 생성
  => 트랜잭션 시작
2) 로그인 여부 확인
3) 글쓰기 내용의 오류 여부 확인
4) 첨부로 업로드된 파일 확인 및 저장
5) 사용자의 입력 내용을 DMBS에 저장
6) 첨부 파일 정보를 DBMS에 저장
7) 저장된 내용 또는 기타 정보를 DBMS에서 조회
8) 게시물 등록에 대한 알림 메일 발송
9) 알림 메일 전송 이력을 DBMS에 저장
  <= 트랜잭션 종료(COMMIT)
  <= 커넥션 바납
10) 처리 완료
```

위 절차 중에서 DMBS의 트랜잭션 처리에 좋지 않은 영향을 미치는 부분을 살펴보자.

- 커넥션 생성과 트랜잭션을 처음부터 시작하지만, 실제로 DB와 상호작용을 하는 것은 5번부터다. 이렇게 커넥션을 소유하는 시간이 길어질수록 사용 가능한 여유 커넥션의 개수는 줄어들 것이다.
- 알림 메일을 발송하는 작업은 트랜잭션에서 제거해야 한다. 외부 알림 메일 서비스에 문제가 생기면 웹 서버뿐만 아니라 DBMS 서버까지 위험해질 수 있다.
- 절대 쪼개져서는 안되는 작업끼리 하나의 트랜잭션으로 묶어야 한다. 4번과 5번은 하나의 트랜잭션으로 묶여야 하며, 단순 조회를 하는 7번은 트랜잭션에 묶일 필요가 없다. 4,5번과 성격이 다른 9번은 별도의 트랜잭션으로 분리하는 게 좋다.

다음은 개선한 구조다.

```
1) 처리 시작
2) 로그인 여부 확인
3) 글쓰기 내용의 오류 여부 확인
  => 커넥션 생성
  => 트랜잭션 시작
4) 첨부로 업로드된 파일 확인 및 저장
5) 사용자의 입력 내용을 DMBS에 저장
6) 첨부 파일 정보를 DBMS에 저장
  <= 트랜잭션 종료(COMMIT)
7) 저장된 내용 또는 기타 정보를 DBMS에서 조회
8) 게시물 등록에 대한 알림 메일 발송
  => 트랜잭션 시작
9) 알림 메일 전송 이력을 DBMS에 저장
  <= 트랜잭션 종료(COMMIT)
  <= 커넥션 반납
10) 처리 완료
```

## MySQL 엔진의 잠금

MySQL에서 사용되는 락은 크게 스토리지 엔진 레벨과 MySQL 엔진 레벨로 나눌 수 있다.

MySQL 엔진 레벨의 락은 모든 스토리지 엔진에 영향을 미치지만, 스토리지 엔진 레벨의 잠금은 스토리지 엔진간 상호 영향을 미치지는 않는다.

MySQL 엔진에는 테이블 락, 메타데이터 락, 네임드 락을 제공한다.

### 글로벌 락

글로벌 락은 FLUSH TABLES WITH READ LOCK 명령으로 획득할 수 있으며, MySQL에서 제공하는 락 가운데 가장 범위가 크다.

한 세션에서 글로벌 락을 획득하면 다른 세션에서 INSERT를 제외한 대부분의 DDL, DML 문장을 실행하는 경우 글로벌 락이 해제될 때까지 기다려야 한다(서버 전체에 영향)

글로벌락은 여러 데이터베이스에 존재하는 MyISAM이나 MEMORY 테이블에 대하 mysqldump로 일관된 백업을 받아야 할 때 사용한다.

하지만 MySQL 서버가 발전해오면서 InnoDB 스토리지 엔진의 사용이 일반화되었고, InnoDB는 태랜잭션을 지원하기 때문에 일관된 데이터 상태를 위해 모든 데이터 변경 작업을 멈출 필요는 없다. 또한 MySQL 8.0부터 InnoDB가 기본 스토리지 엔진됨 가벼운 글로벌 락 필요해짐 -> 백업 락 도입.

```sql
LOCK INSTANCE FOR BACKUP;
UNLOCK INSTANCE;
```

### 테이블 락

- 개별 테이블 단위로 설정되는 락
- 명시적 테이블락:"LOCK TABLES table_name [READ | WRITE]" 명령으로 테이블락 획득 가능
  - 명시적인 테이블 락 일반적으로 사용할 필요 없음. 작업에 영향 미침
- 묵시적 테이블락: MyISAM이나 MEMORY 테이블에 데이터를 변경하는 쿼리 실행 시 발생. MySQL 서버가 데이터가 변경되는 테이블에 잠금을 설정하고 데이터를 변경한 후, 즉시 잠금을 해제하는 형태로 사용.
  - InnoDB 테이블은 레코드 기반의 락 제공하기에 데이터 변경 쿼리로로 인해 묵시적인 테이블락 설정 안됨
    - 자세하게는 테이블락 설정되지만 DML 쿼리에서는 무시되고 DDL 경우에만 영향

### 네임드락

- GET_LOCK() 함수를 이용해 임의의 문자열에 대해 잠금 설정 가능
- 테이블, 레코드 같은 데이터베이스 객체가 아니라 단순히 사용자가 지정한 문자열에 대해 획득하고 반납하는 잠금
- 자주 사용하지 않음
- 데이터베이스 서버 1대에 5대의 웹 서버가 접속하는 상황 같이 여러 클라이언트가 상호 돵기화를 처리해야 할 때 사용한다.

```sql
-- "mylock"이라는 문자열에 대해 잠금 획득
-- 이미 잠금 사용중이라면 2초 동안 대기. (2초 이후 자동 잠금 해제)
SELECT GET_LOCK('mylock' 2);

-- "mylock"이라는 문자열에 대해 잠금이 설정돼 있는지 확인.
SELECT IS_FREE_LOCK('mylock');

-- "mylock"이라는 문자열에 대해 획득했던 잠금을 해제
SELECT RELEASE_LOCK('mylock');

-- 3개 함수 모두 정상적으로 락을 획득하거나 해제한 경우에는 1을
-- 아니면 NULL이나 0을 반환한다.
```

- 또한 많은 레코드에 대해 복잡한 요건으로 레코드를 변경하는 트랜잭션에 유용하게 사용 가능.
- 배치 프로그램처럼 한꺼번에 많은 레코드를 변경하는 쿼리는 자주 데드락의 원인이 되곤한다.
- 동일 데이터를 변경하거나 참조한느 프로그램끼리 분류하여 네임드 락을 걸고 쿼리를 실행하면 아주 간단히 해결 가능

### 메타데이터 락

- 데이터베이스 객체(테이블, 뷰 등)의 이름이나 구조를 변경하는 경우 획득하는 잠금.
- "RENAME TABLE tab_a TO tab_b" 같이 테이블의 이름을 변경하는 경우 자동으로 획득하는 잠금이다.
- 실시간으로 테이블을 바꿔야 하는 요건이 배치 프로그램에서 자주 발생하는데, 다음 예제 참고

```sql
-- 배치 프로그램에서 별도의 임시 테이블(rank_new)에 서비스용 랭킹 데이터 저장

-- 랭킹 배치가 완료되면 서비스용 랭킹 테이블(rank)를 rank_backup으로 백업하고

-- 새로 만들어진 랭킹 테이블(rank_new)을 서비스용으로 대체하고자 하는 경우

RENAME TABLE rank TO rank_backup , rank_new TO rank;
```

- 위와 같이 처리하면 "Table not found 'rand'"와 같은 상황 발생 안함.

## InnoDB 스토리지 엔진 잠금

- InnoDB 슽뢰지 엔진은 MySQL에서 제공하는 잠금과 별개로 슽뢰지 엔진 내부에서 레코드 기반 잠금 방식을 탑재한다.

### InnoDB 스토리지 엔진의 잠금

- 레코드 기반의 잠금 기능 제공
- 잠금 정보가 상당히 작은 공간으로 관리되기에 레코드 락이 페이지 락으로, 또는 테이블 락으로 레벨업 되는 경우(락 에스컬레이션)는 없다.
- 상용 DBMS와 다르게 레코드 락뿐 아니라 레코드와 레코드 사이의 간격을 잠그는 갭(GAP)락이 존재
