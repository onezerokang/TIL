# 스키마 디자인

몽고디비의 스키마를 설계할 때 가장 중요한 부분은 데이터를 도큐먼트에 내장할 것인지, 다른 도큐먼트를 참조할 것인지를 결정하는 것이다.
데이터를 내장하는 경우 임베디드 방식, 참조할 경우 레퍼런스 방식이라 한다.
임베디드 방식은 한번의 쿼리로 데이터를 가져오기에 조회(Read)에 유리하고 레퍼런스 방식은 데이터를 생성, 수정, 삭제하는데 유리하다.

## 임베디드 방식

임베디드 방식은 관계를 갖는 데이터를 단일 도큐먼트에 포함하여 저장하는 방식이다.

```js
const mongoose = require("mongoose");
const { Schema } = mongoose;

const studentSchema = new Schema({
  name: String,
  class: [
    { name: "자료구조", teacher: "조구자" },
    { name: "알고리즘", teacher: "고알리" },
  ],
});
```

### 장점

- 데이터가 내장 되어 있기 때문에 조회 성능이 좋다.
- 데이터 관리가 직관적이고 쿼리가 단순하다.

### 단점

- 데이터 중복은 도큐먼트 구조를 단순화하고 조회 성능을 향상하나 데이터 불일치가 발생할 수 있다.
- 도큐먼트에 포함하는 데이터가 증가할수록 도큐먼트의 크기가 증가하여 디스크 I/O 시 성능 저하 발생 및 도큐먼트의 최대 크기를 초과 시 저장이 불가능할 수 있다.
- 데이터의 관계가 복잡하거나 계층 구조를 갖는 경우 관리가 어렵다.

### 예시

아래는 학생과 수업 데이터를 임베디드 방식으로 저장한 예시 도큐먼트다.

```js
{
  name: "김옥자",
  class:[
    { name: "자료구조", teacher: "조구자" },
    { name: "알고리즘", teacher: "고알리" }
  ]
}
{
  name: "이민관",
  class:[
    { name: "자료구조", teacher: "조구자" },
  ]
}
{
  name: "장수철",
  class:[
    { name: "자료구조", teacher: "조구자" },
    { name: "클린코드", teacher: "마틴" }
  ]
}
```

이렇게 데이터를 내장하여 저장하면 한번의 쿼리로 학생과 수업 모두를 불러올 수 있어 읽기 성능은 좋아지지만 쓰기, 수정, 삭제에는 좋지 않다.

예를들어 '자료구조' 수업의 이름이 '리팩터링'으로 변경됐을 때 '자료구조' 수업을 가지고 있는 기존 데이터를 모두 변경해야 한다. 만약 이 과정에서 장애가 발생한다면 일부 데이터만 정상적으로 변경되고, 나머지 데이터는 변경되지 않는 '데이터 불일치'가 발생할 수 있다.

### 권장 업무

- 조회 성능이 중요하고 데이터 중복에 따른 데이터 불일치 문제가 발생하지 않는 업무
- 업데이트가 과도하게 발생하지 않는 업무
- 내장되는 서브 도큐먼트가 100개 미만인 업무

## 레퍼런스 방식

레퍼런스 방식은 도큐먼트에 관계를 갖는 다른 도큐먼트의 식별자를 참조키로 저장하여 애플리케이션에서 populate 하는 방식이다.
이때 참조키를 어떤 도큐먼트에 저장할지에 따라 3가지 참조방식으로 나뉜다.

_아래 설명에서 1:N 관계일 때 1을 부모 도큐먼트, N을 자식 도큐먼트로 칭하겠습니다._

### 자식 참조

부모 도큐먼트에 자식 도큐먼트의 식별자를 참조키로 저장하는 방식이다. 부모 도큐먼트에서 자식 도큐먼트를 쉽게 찾을 수 있다. 다만 자식 도큐먼트를 추가, 삭제 할 때마다 부모 도큐먼트를 업데이트 해줘야해서 성능 문제가 발생할 수 있으며 자식 도큐먼트가 많은 경우 도큐먼트의 최대 크기를 초과할 수 있다.

자식 도큐먼트의 개수가 100개 이상, 1000개 미만인 경우 적합하다.

```js
const mongoose = require("mongoose");
const { Schema } = mongoose;
const { ObjectId } = Schema.Types;

const studentSchema = new Schema({
  name: String,
  class: [{ ref: "Class", type: ObjectId }],
});

const classSchema = new Schema({
  name: String,
  teacher: String,
});
```

### 부모 참조

자식 도큐먼트에서 관계를 갖는 부모 도큐먼트의 식별자를 참조키로 저장하는 방식이다.
자식 도큐먼트에서 부모 도큐먼트를 쉽게 찾을 수 있으며 자식 도큐먼트의 추가 및 삭제로 인한 부모 도큐먼트의 업데이트가 없다.
자식 참조와 비교했을 때 부모 도큐먼트와 관계를 갖는 모든 자식 도큐먼트 조회 시 소요시간이 증가할 수 있다.

자식 도큐먼트의 개수가 1000개 이상인 경우 적합하다.

```js
const mongoose = require("mongoose");
const { Schema } = mongoose;
const { ObjectId } = Schema.Types;

const studentSchema = new Schema({
  name: String,
});

const classSchema = new Schema({
  name: String,
  teacher: String,
  student: {
    ref: "Student",
    type: ObjectId,
  },
});
```

### 상호 참조

관계를 갖는 부모 도큐먼트와 자식 도큐먼트가 각각 서로의 식별자를 참조키로 저장하는 방식이다.
부모 도큐먼트에서 자식 도큐먼트를 찾거나 자식 도큐먼트에서 부모 도큐먼트를 쉽게 찾을 수 있으나 다른 방식에 비해 참조 정보 관리가 어렵다.
부모 도큐먼트와 자식 도큐먼트 모두 업데이트가 과도하게 발생할 수 있으며 참조 정보 관리 부주의 시 데이터 정합성 문제가 쉽게 발생할 수 있다.
데이터 변경이 적고 부모 도큐먼트와 자식 도큐먼트가 서로를 빈번하게 참조하는 업무에 적합하다.

```js
const mongoose = require("mongoose");
const { Schema } = mongoose;
const { ObjectId } = Schema.Types;

const studentSchema = new Schema({
  name: String,
  class: [{ ref: "Class", type: ObjectId }],
});

const classSchema = new Schema({
  name: String,
  teacher: String,
  student: {
    ref: "Student",
    type: ObjectId,
  },
});
```

## 컬렉션

컬렉션은 관계형 데이터베이스의 테이블과 대응하는 개념으로 용도가 같거나 유사한 도큐먼트들의 그룹을 묶는 단위다. 컬렉션은 4가지 종류로 일반 컬렉션, 캡드 컬렉션, TTL 컬렉션, 시스템 컬렉션으로 나뉜다.

### 일반 컬렉션

가장 일반적으로 사용되는 컬렉션으로, 아래에 나올 컬렉션이 아니라면 모두 일반 컬렉션이다.

### 캡드 컬렉션

캡드 컬렉션은 고정된 크기를 갖는 컬렉션으로 높은 성능의 로깅을 위해 설계됐다.

```
> db.createCollection(name, {capped: <Boolean>, size: <number>, max <number>} )
```

- `name`: 컬렉션의 이름을 입력한다.
- `size`: 최대 저장크기로 단위는 Byte다.
- `max`: 저장할 도큐먼트의 최대 개수를 지정한다.

도큐먼트 추가 시 디스크 공간이 없는 경우 가장 오래된 도큐먼트부터 덮어쓰기 한다.
오래된 데이터를 수동으로 삭제하는 작업을 없애 데이터 관리가 편리히다.
로깅을 위해 만들어져 사용자가 임의로 삭제하거나 업데이트 할 수 없다.

### TTL 컬렉션

TTL 컬렉션은 특정 시간이 경과한 도큐먼트를 자동으로 삭제하는 컬렉션으로 TTL 인덱스에 의해 지원되는 기능이다. 아래는 도큐먼트 생성 1시간 후 제거되는 TTL 인덱스를 설정하는 방법이다.

```
db.collection.createIndex({ expireAfterSeconds : 3600 })
```

TTL 인덱션의 제한 사항은 다음과 같다.

- `_id` 필드에 적용할 수 없다.
- 캡드 컬렉션에 사용할 수 없다.
- 복합(compound)인덱스를 지원하지 않는다.
- 이미 존재하는 인덱스의 `expireAfterSeconds` 값을 `createIndex()` 명령어로 변경할 수 없다. 이미 존재하는 인덱스의 값을 변경하려면 인덱스를 삭제하고 다시 만들어야 한다.

### 시스템 컬렉션

몽고DB 내부에서 사용되는 컬렉션으로 `<database>.system.*` 네임스페이스를 사용한다.

## 스키마 디자인 절차

### 요구사항 수집

사용자가 시스템에서 요구하는 정보를 수집한다.

### 업무 데이터 분석

업무 데이터의 발생 규칙과 관계 및 서비스에 대한 정보를 분석한다.

### 도큐먼트 설계

분석된 업무 데이터를 기반으로 애플리케이션의 처리 방안과 CRUD 성능 등을 고려하여 도큐먼트를 설계한다.

### 컬렉션 설계

컬렉션명을 부여하고, 적절한 컬렉션 종류 선택한다.

## 스키마 디자인의 핵심질문

다음과 같은 질문을 통해 적합한 스키마 구조를 디자인할 수 있다.

- 1:N 관계에서 N < 100을 충족하는가? => 임베디드 방식
- 조회 비율이 생성, 업데이트, 삭제에 비해 높은가? => 임베디드 방식
- 1:N 관계에서 N을 반드시 1과 함께 호출하는가? => 임베디드 or 자식 참조 방식
- 1:N 관계에서 100 < N < 1000을 충족하는가? => 자식 참조 방식
- 1:N 관계에서 1000 < N을 충족하는가? -> 부모 참조 방식
- 1:N 관계에서 다양한 조건으로 N을 탐색하는가? => 부모 참조 방식
- 업데이트가 자주 발생하는가? => 부모 참조 방식

## 마무리

스키마 디자인을 처음부터 완벽하게 할 필요는 없다. 몽고디비는 여러 구조를 만들어보고 수정할 수 있기에
어떻게 설계해야 할지 애매할 때는 일단 관계형으로 디자인하고
나중에 성능과 상황을 고려하여 수정하면 된다.
