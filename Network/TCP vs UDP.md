# TCP vs UDP

TCP(Transmission Control Protrol)는 가장 대표적인 통신 방법으로 전송 제어 프로토콜이라고 불린다.
Internet Protocol로만 통신하면 발생할 수 있는 비 연결성, 비 신뢰성, 프로그램 구분등을 보완한 전송 규약이다.
TCP는 데이터의 신뢰성을 보장한다.

UDP(User Datagram Protocol)는 TCP의 3-way-handshake를 하지 않거나 순서 정보등이 포함되어 있지 않아 데이터 전송을 보장하지 않는다.
하지만 TCP보다 가볍고 빠르기 때문에 전송이 꼭 보장되어야 할 필요가 없거나, 더 최적화하려고 할 때 사용된다.

## IP(Internet Protocol)

인터넷 망에서 데이터를 전송하는 규약을 IP(Internet Protocol)이라고 한다.

클라이언트와 서버는 각각 IP 주소를 부여 받고, IP 주소에 패킷이라는 통신 단위로 데이터 주소를 전달한다.

하지만 IP는 다음과 같은 한계를 가지고 있어, TCP와 함께 사용된다.

- **비 연결성**: 패킷을 받을 대상이 없거나 서비스 불능 상태여도 패킷 전송(서버가 꺼져있다면?)
- **비신뢰성**: 중간에 패킷이 유실되거나, 순서대로 오지 않는다면?
  - 중간에 패킷이 사라지면? -> 거쳐서 오는 서버 중 하나에 문제가 발생하면? 유실될 수 있다(케이블이 짐승에 의해 끊김)
  - 패킷이 순서대로 안오면? -> 전송할 데이터가 커지면 패킷을 쪼개는데, 순서가 다르게 올 수 있다.
- **프로그램 구분**: 같은 IP를 사용하는 장비에서 통신하는 애플리케이션이 둘 이상이면?

이런 Internet Protocol의 한계를 보완하는 것이 TCP다.

## TCP의 특징

전송 제어 프로토콜(Transmission Control Protocol)

- 연결 지향 - TCP 3 way handshake(가상 연결): 연결되었는지를 확인하고 메시지를 보낸다.
- 데이터 전달 보증: 패킷이 중간에 유실되면 알 수 있다.
- 순서 보장

신뢰할 수 있는 프로토콜이다.

전송정보, 검증정보, 순서정보가 포함되어있다.

### TCP 3 way handshake

- SYN: 접속 요청
- ACK: 요청 수락

1. 클라 -> 서버로 SYN 메시지 보낸다.
2. 서버 -> 클라 SYN + ACK 메시지 보낸다.
3. 클라 -> 서버: ACK 메시지를 보낸다.
4. 데이터를 전송한다.

서버가 꺼져있을 경우 SYN 메시지를 보냈을 때 응답을 받을 수 없다. 따라서 메시지를 전송하지 않게 된다.

요즘은 최적화가 돼서 3번 ACK를 보낼 때 데이터도 같이 전송한다.

### 데이터 전달 보증

데이터를 전송하면, 데이터를 잘 받았다는 메시지를 보내준다.

### 순서 보장

패킷1, 패킷2, 패킷3 순서로 전송

하지만 1, 3, 2 순서로 받았다면 다 버리고 패킷 2부터 다시 보내라고 한다.

물론 서버에서 더 최적화하 수 있다.

## UDP

User Datagram Protocol

UDP는 기능이 없다. 하얀 도화지에 비유한다. 데이터 전달 보증 x, 순서 보장 x, IP와 거의 같다. 다만 포트가 추가되었다.

포트는 하나의 IP에서 여러 애플리케이션이 실행중일 때 올바른 곳으로 보내기 위한 주소다. 그리고 체크섬 정도만 추가.

장점: 3 way handshaking을 하면 시간이 발생한다. 데이터량도 크고, 전송 속도도 빠르게 만들기 어렵다. TCP는 손을 못댄다. 하지만 UDP는 더 최적화 할 수 있다. 그 위에다 애플리케이션 레벨에서 무언가를 만들면 된다.

옛날에는 영상 보낼 때 UDP 사용한다고 했지만, SYN, ACK 과정등을 더 최적화 하기 위해 HTTP3는 UDP를 사용하고 있다.
