# TCP vs UDP

## 1. 개요

TCP(Transmission Control Protrol)는 가장 대표적인 통신 방법으로 전송 제어 프로토콜이라고 불린다.
Internet Protocol로만 통신하면 발생할 수 있는 비 연결성, 비 신뢰성, 프로그램 구분등을 보완한 전송 규약이다.

UDP(User Datagram Protocol)는 IP에 포트와 체크섬 정도만 추가된 프로토콜이라 데이터 전송을 보장하지 않는다.
하지만 TCP보다 가볍고 빠르기 때문에 전송이 꼭 보장되어야 할 필요가 없거나, 더 최적화하려고 할 때 사용된다(실제로 HTTP/3부터는 UDP를 사용한다).

## 2. IP(Internet Protocol)

인터넷 망에서 데이터를 전송하는 규약을 IP(Internet Protocol)이라고 한다.

클라이언트와 서버는 각각 IP 주소를 부여 받고, IP 주소에 패킷이라는 통신 단위로 데이터 주소를 전달한다.

하지만 IP는 다음과 같은 한계를 가지고 있어, TCP와 함께 사용된다.

- **비 연결성**: 패킷을 받을 대상이 없거나 서비스 불능 상태여도 패킷을 전송한다.
- **비신뢰성**: 도중에 패킷이 유실되거나, 패킷이 순서대로 도착하지 않을 수 있다.
- **프로그램 구분**: 같은 IP를 사용하는 장비에서 통신하는 애플리케이션이 둘 이상일 경우 어떤 애플리케이션에 데이터를 보내야 하는지 알 수 없다.

## 3. TCP(Transmission Control Protocol)

IP로 인해 데이터가 목적지까지 도착했다면, TCP는 규칙에 의해 올바르게 도착했는지를 따지고 신뢰성을 보장한다.

- **TCP 핸드셰이크**: 연결을 맺고 끊을 때 패킷을 주고 받으며, 서로 연결할 준비가 되었는지(3-way handshake), 끊을 준비가 되었는지(4-way handshake) 확인한다.
- **데이터 전달 보증**: 서버측에서 데이터를 받았다면, 잘 받았다고 응답한다.
- **순서 보장**: 패킷에 순서정보가 포함되어 있어, 잘못된 순서로 도착할 경우 이를 조정한다.

### 3.1. TCP 핸드셰이크

TCP 핸드셰이크는 연결을 시작하고 끊을 때 패킷을 주고 받으며 통신을 맺고 끊을 준비가 되었는지 확인하는 과정이다.

다음은 TCP 핸드셰이크 과정에서 주고 받는 패킷 플래그의 종류다.

- **SYN**: 접속 요청 시 보내는 플래그
- **ACK**: 패킷을 받은 뒤, 잘 받았다고 알려주는 플래그
- **PSH**: 데이터를 즉시 목적지로 보내라는 플래그
- **FIN**: 접속 종료를 위한 플래그

#### 3.1.1 3-way handshake

연결을 시작할 때 진행되는 3-way handshake 과정은 다음과 같다.

1. 클라이언트가 서버에 SYN 패킷을 보내면서 접속 요청을 보낸다.
2. 서버는 SYN 패킷을 잘 받았고, 자신도 접속 요청을 보내기 위해 SYN/ACK 패킷을 보낸다.
3. 클라이언트는 서버에게 ACK 패킷을 보낸다. 이후 ESTABLISHED 상태가 되어 데이터 통신이 가능하게 된다.

#### 3.1.2. 데이터 통신 과정

3-way handshake가 된 이후 통신하는 과정은 다음과 같다.

1. 클라이언트는 Established 된 상태에서 서버에 데이터를 보낸다.
2. 서버는 잘 전송받았다고 ACK 플래그를 넣어 응답한다.
3. 클라이언트가 서버로부터 ACK를 받지못했을 경우 제대로 송신하지 못한 것으로 판단하고 데이터를 재 전송한다.

#### 3.1.2. 4-way handshake

연결을 끊을 때 진행되는 4-way handshake 과정은 다음과 같다.

1. 클라이언트가 연결 종료를 원할 경우 FIN 패킷을 보낸다.
2. 서버는 FIN 패킷을 받으면 ACK 패킷을 전송한다. 만약 서버가 전송할 데이터가 남아있을 경우 나머지를 모두 전송한다.
3. 서버가 모든 데이터를 클라이언트에게 전송하면, 서버는 클라이언트에게 FIN 패킷을 보낸다.
4. 클라이언트가 FIN 패킷을 받으면, 이를 확인하는 ACK 패킷을 보낸다. 해당 패킷이 서버에 도달하면 연결이 완전히 종료된다.

## 4. UDP(User Datagram Protocol)

UDP는 TCP의 과정을 거치지 않기 때문에 헤더의 정보도 더 적고 속도도 더 빠르다. 따라서 실시간 영상 스트리밍 같이 고용량 데이터를 다루는 곳에 이용된다.

HTTP/3.0부터는 TCP가 아닌 UDP를 기반으로 통신한다.
